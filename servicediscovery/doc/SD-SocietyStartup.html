COUGAAR Service Discovery Installation Guide

 

For Windows NT/2000/XP

 

Initial Service Discovery setup (one-time tasks):

 

m.     Install Java 1.4.0 or higher.  (http://java.sun.com/)

 

m.     Download and install the Java Web Services Developer Pack (JWSDP) (from http://java.sun.com/webservices/downloads/webservicespack.html).  This contains the Tomcat server software specially configured for deploying web services.

 

m.     Download and install the Systinet WASP UDDI Registry server (http://www.systinet.com/).  Click on download (in red above the search box).  Download WASP UDDI, 4.0 with Service Pack 1.  Note:  The 4.0 package is now only available in the Archives section.  Unzip the WASP UDDI package.  The target directory will be referred to through the parameter WASP_UDDI_HOME.  Installation/configuration instructions:

 

a.                               Run WASP_UDDI_HOME\bin\install.bat

b.                              Answer “install” to install the current version.

c.                               Answer “no” for installation without SSL and JCE/JSSE.

d.                              Select the default security provider (shown inside brackets).

e.                               Open the file WASP_UDDI_HOME\app\uddi\WASP-INF\wasp.xml, find the element <poolSize>, and change its value to be greater than the maximum expected number of simultaneous clients (a value slightly greater than the number of agents in your society is safe).

 

m.     Download and install the PostgreSQL database server.  PostgreSQL is normally run under the Cygwin Unix emulator for Windows.  The following instructions will download and install Cygwin as well as PostgreSQL:

 

a.                               Installing Cygwin (version 1.3.10 or later):

 

(1)                                             Click on the "Install Cygwin now" link on the http://sources.redhat.com/cygwin web page. This downloads setup.exe to your system.

 

(2)                                             Run setup.exe and answer all of the questions.  The following choices are recommended:

(a)                                                    Choose a download source: Install from internet

(b)                                                   Select Root Install Directory: (your choice)

(c)                                                    Select local package directory: (your choice)

(d)                                                   Select your internet connection: Direct connection

(e)                                                    Choose a download site: (your choice)

(f)                                                     Select packages: You are presented with a hierarchical list of software components to download/install.  Clicking on the “+” expands the tree to reveal individual components.  Avoid selecting “All Default” since it will download many megabytes of programs, most of which you won’t need.  To select a component for download, click on the “circling arrows” icon.  When the component has been selected, the word “Skip” will be replaced by the version number of the program you selected.  You will automatically get the contents of the “Base” component.  In addition, choose at least the following:

 

                                                   i.                                                                  Database: postgresql

                                                 ii.                                                                  Shells: bash

                                                iii.                                                                  Utils: bzip2

 

When this process finishes, PostgreSQL will be installed on your system.

 

b.                              Run Cygwin by clicking on the icon on your Desktop or in the Start menu.  Perform all remaining PostgreSQL setup/startup/shutdown functions from within Cygwin.

 

c.                               Installing the Cygwin32 IPC (Inter-Process Communication) package (required):

 

(1)                                             Download this file from UltraLog DocuShare under Working Groups -> Service Discovery   -> Cygipc  (or “version of cygipc that winzip can handle” if you didn’t install bzip2) and store it in your Cygwin install directory.

 

(2)                                             From within Cygwin: unzip IPC with the command:

 

bunzip2 cygipc-1.13-2.tar.bz2  (if you downloaded the .bz2 version)

 

then unpack it (either version) with the command:

 

tar xvf cygipc-1.13-2.tar

 

d.                              PostgreSQL setup (done from within Cygwin):

 

(1)         Prepare the database directory structure with the command:

 

mkdir –p /usr/local/pgsql/data

 

(2)         Determine your home directory by issuing the following two commands:

 

cd

pwd

 

                     The output from the pwd command is your Cygwin home directory.

 

(3)         Check for the file .bash_login in your home directory.  If it does not exist, create it using any editor.  Append the following to this file:

 

PGDATA=/usr/local/pgsql/data

export PGDATA

 

(4)         Set up properties by running the following:

 

source .bash_login

 

 

(5)         Verify that the PGDATA environment variable has been set by entering:

 

      env

 

         This produces a list of the current environment variables.  Check that PGDATA is included and, if not, repeat the two previous steps.

 

(6)         Run IPC emulation:

 

ipc-daemon &

 

(7)         Create a database:

 

      initdb –U postgres

 

         Note that the owner of the database is ‘postgres’ in this case.

 

(8)         Edit the file /usr/local/pgsql/data/pg_hba.conf, so that it contains the following lines in the section headed “Put your actual configuration here”:

 

            local  all   all                                      trust

            host   all   all  127.0.0.1          255.255.255.255  trust

            host   all   all  (your IP address)  255.255.255.0    trust

 

(9)         Edit the file /usr/local/pgsql/data/postgresql.conf to uncomment the tcpip_socket property and set it to “true”:

 

tcpip_socket=true

 

(10)     Start PostgreSQL:

 
postmaster -i &
 

(11)     To check that the PostgreSQL database is installed properly, run the SQL interpreter “psql” (NOTE: do not confuse “psql” with “pgsql”):

 

psql –h localhost template1 postgres

 

                     (“template1” is the default database created in PostgreSQL.)

                     Type ‘\q’ to quit psql.

 
 

e.                               Stopping PostgreSQL:

 

(1)         Run the pg_ctl script to stop the postgres server:

 

pg_ctl stop

 

 

 

f.                                Tips:

 

(1)         To see whether 'ipc-daemon' or 'postmaster' is running, type 'ps' in cygwin or launch the Task Manager in Windows.

 

(2)         To process SQL commands, run the SQL console:

 

psql -h <host_name> <db name> <user name>

 

 

(3)         To check the status of PostgreSQL, run the pg_ctl script:

 

pg_ctl status

 

(4)         If you attempt to start PostgreSQL and get the message that PostgreSQL is already running, yet no process “post*” can be found in the “ps” list, it probably indicates that a previous postgres instance was shutdown incorrectly.  Remove the $PGDATA/postmaster.pid file.

 

g.                               Load the PostgreSQL database.  For this step, please refer to the Database Installation section of the WASP UDDI server installation documentation (WASP_UDDI_HOME\doc\databaseInstallationGuide\index0.html).  See the section entitled “UDDI Database Installation on PostgreSQL (Windows 2000 and Linux)”. 

NOTE: When running the installPostgresqlDB.bat file, accept all the default answers to the questions except: "Do you want to write database properties to WASP UDDI configuration - Wasp UDDI Must Be Down (y/n) [n]" - answer “yes”.

 

 

m.     Configure the WASP UDDI Registry (the PostgreSQL server must be running):

a.                               Start the WASP UDDI Registry server: WASP_UDDI_HOME\bin\serverstart.bat

b.                              Launch a web browser pointed at http://(hostname):(port)/uddi/web?configuration

c.                               Click on the “Registry Administration” button.

d.                              Supply the username “admin” and password “changeit”.  The Configuration page will load.

e.                               Click on the “Registry Configuration” button.

f.                                Select the “Database” tab.  Set the “Database type” item to “PostgreSQL”.  The “Driver” item should be automatically set by WASP to “PostgresqlConnection”.  Set the remaining items to the values you selected during the loading of the PostgreSQL database.  IMPORTANT: Ensure that the “Db name” field contains only the name of the database just created and does not include the path.  For example, if the database name you selected is “uddinode”, ensure that only “uddinode” appears in the “Db name” field, not “C:\waspuddi40/database/uddinode”.  Also, increase the number of database connections to at least 30 and the timeout to at least 20.

g.                               Select the “Account” tab.  Change the default allowances to 1000 for the following structures: Business entity, Business service, Binding template, TModel, Assertion.

h.                               Click “Save”.  After notification that the registry configuration was successfully saved, your WASP UDDI server is ready for use.

i.                                 Return to the “Server Administration” page and click the “Accounts management” button.  Click the “Create account” button and create a user account that you will use for programmatic interaction with the registry server.

 

 

m.     Install and configure Cougaar. 

a.                               Set the COUGAAR_INSTALL_PATH environment variable to your Cougaar install path (CIP).

 

b.                              Edit CIP\servicediscovery\bin\configure-registry.bat and clear-registry.bat to add the correct hostname and port on which the WASP server is running (default values are localhost and 8080, respectively).  Note:  If you are building the service discovery module, you will need to add the %WASP_HOME%\lib\wasp.jar and %WASP_HOME%\dist\uddiclient.jar to the classpath.  You will see these referenced in the configure-registry.bat script.  These do not need to be included in the classpath when running the society.

 

c.                               With the PostgreSQL and WASP servers running, run the configure-registry.bat script.  NOTE: This operation loads the taxonomies (located in CIP\servicediscovery\data\taxonomies) and Tmodels required for service discovery into the UDDI registry.  This must be done prior to running the society; otherwise service discovery will fail. 

 

d.                              If you will be running the Cougaar society from CSMART: edit CIP\servicediscovery\bin\registryArgs.props to add your registry username, password, hostname, and port.

 

e.                               Copy a cougaar.rc file from another Cougaar installation into the CIP\configs\common and update the hostname, db names, userid, and password for the database server to be used.

 

f.                                If your CIP is not C:/cougaar, run the following Perl script from CIP\servicediscovery\data\serviceprofiles:

 

         perl setCougaarInstallPath.pl <your CIP> C:/cougaar

 

This will change all instances of the Cougaar install path in the commercial side .profile.daml files to match yours.  If Perl is not installed on your computer, you can either download it (from http://www.activestate.com) and install it, or manually edit each of the commercial side .profile.daml files that came with the distribution, or copy the files from another location where they have already been updated.

 

g.                               Edit CIP\servicediscovery\data\serviceprofiles\agent-input.txt to update the Cougaar install path to your CIP.  Then, run the following Perl script to create additional military side .profile.daml files:

 

         perl generateDAML.pl agent-input.txt profile-template.txt

     

If you elected not to install Perl on your computer, copy these files from another location where they have already been created.

 

h.                               Edit CIP\servicediscovery\data\common\waspUtil.props to insert the correct registry hostname and port.

 

i.                                 If running the society from CSMART: Go to (CIP)\server\bin\server.props and locate the org.cougaar.config.path property.  Comment out the current value for this property and uncomment the alternate value, which should contain the following path element:

 

               (CIP)\\servicediscovery\\data\\serviceprofiles\;

 

            Additionally, this config path should contain the element

 

               (CIP)\\servicediscovery\\configs\\common;

 

      and it should occur prior to the entry (CIP)\\configs\\common.

 

j.                                Create new Cougaar and, if running from CSMART, Configuration (CSMART) databases in MySQL.

 

k.                              Load the Cougaar database:

(1)         Unzip the two zip files in CIP\dbms\data\mysql

(2)         cd CIP\dbms\data\mysql

(3)         mysql (cougaar db name) < 1ADDomainData.sql

(4)         mysql (cougaar db name) < army.txt

(5)         mysql (cougaar db name) < spares_index.txt

 

l.                                 Load the Configuration (CSMART) database:

(1)         cd CIP\csmart\data\database\scripts\mysql

(2)         load_1ad_mysql.bat cd (userid) (password) (CSMARTSD db name)

(3)         cd CIP\servicediscovery\data\database\scripts\mysql

(4)         load_sd_1ad_mysql.bat (userid) (password) (CSMARTSD db name)

 

n.       If running from CSMART:

(1)         Import the following recipes (found in CIP\servicediscovery\data\csmart\recipe):

ServiceDiscoveryRegistration-export.sql

   ServiceDiscoveryBootstrapping-export.sql

   ServiceDiscoveryMaintenance-export.sql       

   ServiceDiscoveryMatchmaking-export.sql

     

   To import all recipes at once:

                        cd CIP\servicediscovery\bin

            load-recipes.bat (userid) (password) (CSMARTSD db name)

 

        To import individual recipes:

mysql –u (userid) –p(password) (CSMART db name) < (recipe file name)

 

(2)          Edit CIP\csmart\CSMART.bat. The MYCONFIGSPATH definition must include %COUGAAR_INSTALL_PATH%\servicediscovery\configs\common to ensure that the queries referenced by the Service Discovery recipes are accessible.

 

 

 

Example:     SET MYCONFIGPATH=-Dorg.cougaar.config.path="%COUGAAR_INSTALL_PATH%/

csmart/data/common/\; %COUGAAR_INSTALL_PATH%/servicediscovery/configs/common/\;%COUG

AAR_INSTALL_PATH%/configs/minitestconfig/\;"

 

n.       If running from the command line: add the following Service Discovery specific definitions to CIP\bin\setarguments.bat. CIP\servicediscovery\bin\setarguments.bat can be used as a template.

 

(1)         Include both  %COUGAAR_INSTALL_PATH%\servicediscovery\configs\common  and %COUGAAR_INSTALL_PATH%\servicediscovery\data\serviceprofiles in the definition of - Dorg.cougaar.config.path . %COUGAAR_INSTALL_PATH%\servicediscovery\configs\common must be referenced before %COUGAAR_INSTALL_PATH\configs\common to ensure that the Service Discovery domain is used. 

 

Example:

-Dorg.cougaar.config.path=%COUGAAR_INSTALL_PATH%\servicediscovery\configs\common;%COUG

AAR_INSTALL_PATH%\configs\common;%COUGAAR_INSTALL_PATH%\servicediscovery\data\serviceprofiles;

 

(2)         Add the following definitions so that ServiceDiscovery can access the UDDI registry -org.cougaar.servicediscovery.registry.username (UDDI username), org.cougaar.servicediscovery.registry.password (UDDI password), org.cougaar.servicediscovery.registry.queryURL (UDDI query URL), and org.cougaar.servicediscovery.registry.publishURL (UDDI publish URL)

 

Example:

-Dorg.cougaar.servicediscovery.registry.username=ulservices

-Dorg.cougaar.servicediscovery.registry.password=ulservices

-Dorg.cougaar.servicediscovery.registry.queryURL=http://localhost:8080/uddi/inquiry

-Dorg.cougaar.servicediscovery.registry.publishURL=http://localhost:8080/uddi/publishing

 

 

o.                  Configure web services:

 

a.                               Put CIP\servicediscovery\data\soap.war into your Tomcat webapps directory.

 

b.                              Start Tomcat:

 

(JWSDP install dir)\bin\startup.bat

 

c.                               Go to http://<machine name>:<port>/soap/admin/index.html.

 

d.                              Click on the “deploy” link and deploy the following services:

 

(1)         GraingerWebService

(a)          ID = urn:GraingerWebService

(b)         Scope = Application

(c)          Methods= getPartDescription getPartPrice isAvailable

(d)         Provider Class = grainger.GraingerWebService

(e)          click the deploy button on the bottom

 

(2)         NewarkWebService

(a)          ID = urn:NewarkWebService

(b)         Scope = Application

(c)          Methods= getPartDescription getPartPrice isAvailable

(d)         Provider Class = newark.NewarkWebService

(e)          click the deploy button on the bottom

 

e.                               In CIP\servicediscovery\data\servicegroundings:

 

(1)         Modify GraingerWebService.wsdl line number 99 to point to your machine.

(2)         Modify NewarkWebService.wsdl line number 102 to point to your machine.

(3)         Move these wsdl files to your Tomcat webapps\ROOT directory.

 

f.                                Modify Grainger.profile.daml and Newark.profile.daml in CIP\servicediscovery\data\serviceprofiles to point to these wsdl files.

 

 

Startup:

 

1.      Start the PostgreSQL server (from within Cygwin):

               ipc-daemon &

postmaster -i &

 

If you receive an error message indicating that the environment variable PGDATA is not found, enter the following commands (in Cygwin):

 

     cd   [this takes you to your home directory]

     source .bash_login

 

Then repeat the  postmaster –i &  command (shown above).

 

2.      Start the WASP UDDI Registry server:

 

(WASP UDDI install dir)\bin\serverstart.bat

 

3.      Start Tomcat (if it is not already running – see step 7.b. in the “Initial Service Discovery setup” section).

 

 

Running from the command line:

 

1)      IMPORTANT: Prior to each run (except the first on a new WASP installation), be sure to run CIP\servicediscovery\bin\clear-registry.bat to clear out old registrations.  This script takes two mandatory arguments: the registry user account username and password. 

 

2)      Change directory to CIP\servicediscovery\configs\sdtest,  then run

3)    ..\..\..\bin\Node MiniNode

 

4)      Publish oplan when the society quiesces.  CAVEAT:   Society quiescence (the appearance of dots …..) indicates no agent to agent communication, however, there may still be writes to the registry database.  It is recommended that you wait until the registration process has completed by monitoring CPU activity before sending the Oplan.  Send GLS Root when the society quiesces again. 

 

 

Running from CSMART:

 

1.   IMPORTANT: Prior to each run (except the first on a new WASP installation), be sure to run CIP\servicediscovery\bin\clear-registry.bat to clear out old registrations.  This script takes two mandatory arguments: the registry user account username and password. 

 

2.   Use experiment SD-1AD-TRANS-STUB.

 

3.   Add the following recipes:

 

ServiceDiscoveryRegistration

ServiceDiscoveryBootstrapping

ServiceDiscoveryMaintenance

ServiceDiscoveryMatchmaking

 

      IMPORTANT:  Ensure ServiceDiscoveryRegistration is listed first in the experiment.

 

4.   Add Global Command Line Arguments to the experiment as follows:

a.       Right click on experiment name, select Build

b.      Select the Configuration tab

c.       Right click on the host name in the Hosts pane (left-most pane)

d.      Click on Global Command Line Arguments

e.       Click the Read From File button, then select CIP\servicediscovery\bin\registryArgs.props.

 

5.   Close the Experiment Builder and start the experiment.  Service providers will register with the UDDI registry.  Publish the oplan when the society quiesces.  CAVEAT:  Society quiescence (the appearance of dots …..) indicates no agent to agent communication, however, there may still be writes to the registry database.  It is recommended that you wait until the registration process has completed by monitoring CPU activity before sending the Oplan. 

 

The registry maintains an audit table.  This table stores every write/delete transaction that occurs on user accounts.  This table can get rather large, so you may want to periodically clear it out.

Example:

~> psql -h localhost -U uddiuser uddinode

Welcome to psql 7.3, the PostgreSQL interactive terminal.

 

Type:  \copyright for distribution terms

       \h for help with SQL commands

       \? for help on internal slash commands

       \g or terminate with semicolon to execute query

       \q to quit

 

uddinode=# delete from accountaudit where authorizedname = 'myusername';
